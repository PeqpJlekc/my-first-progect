Процедура ревьюирования кода проекта

​1. Этапы процесса
​Процесс проверки кода (Code Review) в нашем проекте проходит следующие стадии:
​Подготовка: Автор кода проверяет свои изменения на соответствие заданию, прогоняет тесты (если они есть) и создает Pull Request (PR).
​Анализ: Назначенный ревизор изучает код, проверяет его логику, структуру и соответствие критериям качества.
​Обратная связь: Ревизор оставляет комментарии к конкретным строкам кода. Если есть ошибки, PR помечается как «Требует изменений» (Changes requested).
​Повторное ревью: Автор исправляет замечания и пушит обновления. Ревизор проверяет исправления и, если всё в порядке, подтверждает слияние (Approve).
​2. Критерии проверки
​Ревизор должен оценивать код по следующим пунктам:
​Чистота кода: Отсутствие лишних комментариев, понятные названия переменных и функций.
​Работоспособность: Код должен выполнять поставленную задачу без ошибок.
​Безопасность: Отсутствие «захардкоженных» паролей или уязвимых мест.
​Документирование: Наличие описания новых функций в файле README.md.
​3. Назначение ревизоров
​Кто может быть ревизором: Любой участник команды, не являющийся автором данного кода.
​Приоритет: Ревизором назначается человек, наиболее знакомый с данной частью функционала (например, если правился файл main.py, ревью делает его основной разработчик).
​Автоматизация: Для небольших проектов ревизор может назначаться по очереди (Round-robin).
